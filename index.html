1. Overview: Why Align Language Models?

Pre-trained language models can generate fluent text, but they don't inherently know what humans want. RLHF (Reinforcement Learning from Human Feedback) bridges this gap by fine-tuning a model to prefer outputs that humans rate highly.

The three algorithms below represent the main approaches used in production today:

  • PPO — full RL pipeline with a learned reward model (InstructGPT, early ChatGPT)
  • DPO — supervised shortcut that skips the reward model (Llama 2/3, Zephyr)
  • GRPO — simplified RL with group-relative scoring, no critic network (DeepSeek-R1)

All code examples below come from rlhf_algorithms.py, a pure-NumPy educational implementation you can run with just pip install numpy.

2. PPO — Proximal Policy Optimization

Schulman et al., 2017 • Used by InstructGPT & early ChatGPT

Pipeline

Prompt → Policy generates response → Reward model scores it → PPO clips & updates

Key Idea

Train a separate reward model on human preferences, then use it to score policy outputs. The policy is updated via a clipped surrogate objective that prevents catastrophically large updates.

Core Equations

Probability Ratio
$$r_t(\theta) = \frac{\pi_\theta(a_t \mid s_t)}{\pi_{\theta_\text{old}}(a_t \mid s_t)}$$
Clipped Surrogate Objective (maximize)
$$L^{\text{CLIP}}(\theta) = \mathbb{E}_t\!\Big[\min\!\Big(r_t(\theta)\,\hat{A}_t,\;\text{clip}\!\big(r_t(\theta),\, 1{-}\epsilon,\, 1{+}\epsilon\big)\,\hat{A}_t\Big)\Big]$$

When the advantage \(\hat{A}_t > 0\), the ratio is clipped at \(1 + \epsilon\) so the policy can't move too far in the "good" direction. When \(\hat{A}_t < 0\), it's clipped at \(1 - \epsilon\). The \(\min\) selects the pessimistic bound.

KL Penalty (keeps policy near the reference SFT model)
$$D_{\text{KL}}(\pi_\theta \| \pi_\text{ref}) \approx \mathbb{E}\!\Big[\exp\!\big(\log\pi_\theta - \log\pi_\text{ref}\big) - \big(\log\pi_\theta - \log\pi_\text{ref}\big) - 1\Big]$$
Combined Loss
$$\mathcal{L}_{\text{PPO}} = -L^{\text{CLIP}}(\theta) + \beta \cdot D_{\text{KL}}(\pi_\theta \| \pi_\text{ref})$$

Annotated Code

def ppo_update(policy, states, actions, old_log_probs, advantages,
               ref_log_probs, lr=0.01, clip_eps=0.2, kl_coeff=0.1):

    new_log_probs = policy.log_probs(states, actions)
    grad_lp = policy.grad_log_prob(states, actions)

    # ratio = pi_new / pi_old  (in log space for stability)
    ratio = np.exp(new_log_probs - old_log_probs)

    # Clipped surrogate: take the pessimistic bound
    surr1 = ratio * advantages
    surr2 = np.clip(ratio, 1.0 - clip_eps, 1.0 + clip_eps) * advantages
    use_unclipped = (surr1 <= surr2).astype(float)

    # Policy gradient (ascent direction)
    ppo_grad = (use_unclipped * advantages * ratio)[:, None, None] * grad_lp
    ppo_grad = ppo_grad.mean(axis=0)

    # KL penalty to stay near the reference (SFT) model
    log_diff = new_log_probs - ref_log_probs
    kl_grad = (kl_coeff * (np.exp(log_diff) - 1.0))[:, None, None] * grad_lp
    kl_grad = kl_grad.mean(axis=0)

    # Ascend on objective, descend on KL
    policy.update(lr * (ppo_grad - kl_grad))
Why clipping? Without it, a single batch with high advantage could push the policy far from the old policy, causing training instability. The clip acts as a trust region.

3. DPO — Direct Preference Optimization

Rafailov et al., 2023 • Used by Llama 2/3, Zephyr

Pipeline

Preference pairs (yw, yl) → One supervised loss → Updated policy

Key Idea

DPO shows that you can skip the reward model entirely. Given human preference pairs (chosen response \(y_w\) vs. rejected \(y_l\)), DPO derives a closed-form loss from the Bradley-Terry preference model. The loss directly increases the gap between chosen and rejected log-probabilities, relative to a frozen reference model.

Core Equations

DPO Loss
$$\mathcal{L}_{\text{DPO}}(\theta) = -\mathbb{E}\!\Big[\log\sigma\!\Big(\beta\Big[\underbrace{\log\frac{\pi_\theta(y_w)}{\pi_\text{ref}(y_w)}}_{\text{chosen reward}} - \underbrace{\log\frac{\pi_\theta(y_l)}{\pi_\text{ref}(y_l)}}_{\text{rejected reward}}\Big]\Big)\Big]$$

The \(\sigma(\cdot)\) is the sigmoid function. The loss is minimized when the policy assigns much higher probability to the chosen response (relative to the reference) than to the rejected one.

Gradient
$$\nabla_\theta \mathcal{L}_{\text{DPO}} = -\beta\,\mathbb{E}\!\Big[\underbrace{\big(1 - \sigma(\beta\, m)\big)}_{\text{weight}}\;\big(\nabla_\theta \log\pi_\theta(y_w) - \nabla_\theta \log\pi_\theta(y_l)\big)\Big]$$

where \(m = \log\frac{\pi_\theta(y_w)}{\pi_\text{ref}(y_w)} - \log\frac{\pi_\theta(y_l)}{\pi_\text{ref}(y_l)}\) is the margin.

The weight \(1 - \sigma(\beta\, m)\) is large when the policy hasn't yet learned the preference (low margin), and shrinks as the model gets it right. This makes DPO self-pacing.

Annotated Code

def dpo_update(policy, ref_policy, states, chosen_actions,
               rejected_actions, lr=0.01, beta=0.1):

    # Log-probs under current and reference policies
    lp_chosen  = policy.log_probs(states, chosen_actions)
    lp_rejected = policy.log_probs(states, rejected_actions)
    ref_chosen  = ref_policy.log_probs(states, chosen_actions)
    ref_rejected = ref_policy.log_probs(states, rejected_actions)

    # Margin: how much more does policy prefer chosen vs rejected,
    #         compared to the reference model?
    margin = (lp_chosen - ref_chosen) - (lp_rejected - ref_rejected)
    logits = beta * margin

    # Bradley-Terry loss: -log sigma(beta * margin)
    sigmoid = 1.0 / (1.0 + np.exp(-logits))
    loss = -np.log(sigmoid + 1e-10).mean()

    # Gradient: push chosen up, rejected down
    grad_chosen  = policy.grad_log_prob(states, chosen_actions)
    grad_rejected = policy.grad_log_prob(states, rejected_actions)
    weight = -beta * (1.0 - sigmoid)
    grad = (weight[:, None, None] * (grad_chosen - grad_rejected)).mean(axis=0)

    policy.update(-lr * grad)
Why is DPO popular? It's dramatically simpler than PPO — no reward model to train, no RL loop, no critic. Just one loss function applied to preference pairs. The downside: you need high-quality preference data.

4. GRPO — Group Relative Policy Optimization

Shao et al., 2024 • Used by DeepSeek-R1

Pipeline

Prompt → Sample G responses → Score all G → Normalize within group → PPO-style update

Key Idea

For each prompt, sample a group of \(G\) responses. Score them all with a reward function, then normalize rewards within the group to get advantages. This eliminates the need for a critic/value network — the group statistics are the baseline.

Core Equations

Group-Relative Advantage (the key insight)
$$\hat{A}_i = \frac{r_i - \text{mean}(\{r_1, \dots, r_G\})}{\text{std}(\{r_1, \dots, r_G\})}$$

Each response is compared only to its siblings in the group, not to an absolute baseline. This is simple, robust, and requires no learned value function.

GRPO Objective (same clip structure as PPO)
$$\mathcal{L}_{\text{GRPO}}(\theta) = -\frac{1}{G}\sum_{i=1}^{G}\min\!\Big(r_i(\theta)\,\hat{A}_i,\;\text{clip}\!\big(r_i(\theta), 1{-}\epsilon, 1{+}\epsilon\big)\,\hat{A}_i\Big) + \beta\, D_{\text{KL}}(\pi_\theta \| \pi_\text{ref})$$
KL Divergence (same approximation as PPO)
$$D_{\text{KL}} \approx \frac{1}{G}\sum_{i=1}^{G}\Big[\exp\!\big(\log\pi_\theta(a_i) - \log\pi_\text{ref}(a_i)\big) - \big(\log\pi_\theta(a_i) - \log\pi_\text{ref}(a_i)\big) - 1\Big]$$

Annotated Code

def grpo_update(policy, ref_policy, prompt, group_size=8,
                reward_fn=None, lr=0.01, clip_eps=0.2, kl_coeff=0.04):

    states = np.tile(prompt, (group_size, 1))  # repeat prompt G times

    # Step 1: sample G responses from current policy
    actions = policy.sample(states)

    # Step 2: score each response
    rewards = reward_fn(actions)

    # Step 3: GROUP-RELATIVE normalization → advantages
    # This is the key GRPO insight: no critic needed!
    mean_r = rewards.mean()
    std_r  = rewards.std() + 1e-8
    advantages = (rewards - mean_r) / std_r

    # Step 4: PPO-style clipped update
    old_log_probs = policy.log_probs(states, actions)
    new_log_probs = policy.log_probs(states, actions)
    grad_lp = policy.grad_log_prob(states, actions)

    ratio = np.exp(new_log_probs - old_log_probs)
    surr1 = ratio * advantages
    surr2 = np.clip(ratio, 1.0 - clip_eps, 1.0 + clip_eps) * advantages
    use_unclipped = (surr1 <= surr2).astype(float)

    policy_grad = (use_unclipped * advantages * ratio)[:, None, None] * grad_lp
    policy_grad = policy_grad.mean(axis=0)

    # KL penalty vs reference
    ref_log_probs = ref_policy.log_probs(states, actions)
    log_diff = new_log_probs - ref_log_probs
    kl_grad = (kl_coeff * (np.exp(log_diff) - 1.0))[:, None, None] * grad_lp
    kl_grad = kl_grad.mean(axis=0)

    policy.update(lr * (policy_grad - kl_grad))
Why GRPO for reasoning? Math and code problems have verifiable answers. A simple rule-based reward (correct/incorrect) combined with group-relative ranking works remarkably well — no expensive reward model training needed.

5. Side-by-Side Comparison

PPO DPO GRPO
Reward Signal Learned reward model (separate NN) None — implicit in preference pairs Rule-based or learned reward
Critic / Value Net Yes — needs a value network No No — uses group-relative normalization
Training Style RL loop: sample → score → clip → update Supervised: learn from (chosen, rejected) pairs RL loop (simpler): sample G → rank → clip → update
Data Needed Prompts + reward model Preference pairs \((y_w, y_l)\) Prompts + reward function
Complexity High (RM + critic + policy + ref) Low (one loss function) Medium (no critic, but still RL loop)
Best For General alignment with nuanced rewards When you have quality preference data Verifiable tasks (math, code)
Used By InstructGPT, early ChatGPT Llama 2/3, Zephyr DeepSeek-R1

Key Takeaway

  • PPO = full RL pipeline, most general, most complex
  • DPO = supervised shortcut, elegant but needs preference pairs
  • GRPO = simplified RL, no critic, great for verifiable rewards
